Asset Lifecycle Management (ALM) Platform

This document is the single source of truth for the ALM platform, covering product requirements, UX rules, system architecture, database design, delivery planning, and engineering execution.

Start Date: 2 February
Target Audience: Product, Engineering, Architecture, Security, Leadership

2. Product Requirements (PRD)

Objective

Build a multi-tenant, audit-grade Asset Lifecycle Intelligence platform integrating IoT, operations, warranty, and finance into a single system of record.

In Scope

  • Multi-tenant organization model (DB-enforced)
  • Workspace hierarchy (max 6 levels)
  • Asset lifecycle tracking
  • Warranty & usage analytics
  • Rule-based alerts and recommendations
  • Asset transfer workflows

Out of Scope (V1)

  • Custom BI builder
  • Marketplace transactions
  • ML model training pipelines

3. UX & UI Guidelines

Information Architecture

Organization
 └── Workspace Tree
      └── Asset List
           └── Asset Detail
                ├── Health
                ├── Usage
                ├── Warranty
                ├── Events
                └── History

UX Rules

  • Context-first navigation (Org → Workspace → Asset)
  • History is immutable and always visible
  • Charts must explain actions, not just data
  • Red indicates irreversible risk only

4. Technical Architecture

IoT Devices / BMS
      ↓
MQTT / HTTPS
      ↓
Stream Layer (Kafka / Kinesis)
      ↓
Telemetry DB (TimescaleDB)
      ↓
Rule Engine (pg_cron)
      ↓
Core ALM DB (PostgreSQL + RLS + ltree)
      ↓
API Layer (Fastify)
      ↓
Web Dashboard / Partner APIs

Key Principles

  • Database-enforced isolation (RLS)
  • Immutable audit trails
  • Executable rule engine
  • Telemetry isolated from transactions

6. Database Schema (Core)

Core Tables

  • organizations
  • users
  • products (global)
  • assets
  • workspaces (ltree)
  • workspace_users
  • workspace_assets
  • asset_status
  • asset_transfers
  • executable_rules
  • rule_execution_log

All tenant-sensitive tables enforce Row Level Security (RLS).

9. Engineering Guide

First 30 Days

  • Understand domain invariants (asset, workspace, transfer)
  • Never bypass RLS in queries
  • Prefer DB constraints over app logic
  • Preserve history; never overwrite

Golden Rules

  • No cross-org joins
  • No recursive hierarchy queries (use ltree)
  • No JSON-based business rules
  • No silent automation without audit logs

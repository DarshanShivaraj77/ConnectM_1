2. Product Requirements (PRD)

In Scope

  • Multi-tenant organization model (DB-enforced isolation)
  • Workspace hierarchy (max 6 levels)
  • Asset lifecycle tracking (Inventory → Operations → Quarantine → Decommission)
  • Warranty & usage analytics with fraud detection
  • Rule-based alerts and recommendations
  • Asset transfer workflows between organizations
  • Battery passport and ESG compliance
  • Multi-stakeholder collaboration (OEM, Financier, Fleet, 3PL)

Out of Scope (V1)

  • Custom BI builder
  • Marketplace transactions
  • ML model training pipelines
  • Predictive maintenance (deferred to V2)

3. Product Level Drill-Down Logic

The core logic follows a hierarchical template system. When a product is onboarded, it defines the "DNA" for all assets under it.

Hierarchy: Product Type (Industry Level) → OEM (Manufacturer) → Model → Variant (Specific Specs)

Simulated Asset Path Example

  • Type: 4W Passenger Car
  • OEM: Tata
  • Model: Nexon
  • Variant: ZXI EV
  • Slots: Battery Pack, Traction Motor, IoT Gateway

Slot Provisioning Philosophy

Intent is not to onboard every bolt, but to map critical sub-components that stream IoT data for lifecycle monitoring (e.g., Battery SoC, Motor RPM, Temperature sensors).

-- Products are NOT scoped to organizations
ALTER TABLE products DROP CONSTRAINT IF EXISTS products_org_id_fkey;

-- Assets link to organizations
CREATE TABLE assets (
    asset_id UUID PRIMARY KEY,
    product_id UUID REFERENCES products(product_id),
    current_org_id UUID REFERENCES organizations(org_id),
    -- Transfer history preserved
    created_at TIMESTAMP DEFAULT NOW()
);

Key Relationships

  • Products are Global: Once approved, products are available to all organizations
  • Assets belong to One Org: But can be transferred with full audit trail
  • Workspaces use ltree: For efficient hierarchical queries (max 6 levels)
  • Physical vs Logical: Assets have one physical location but multiple stakeholder views

14. Security & Row Level Security (RLS)

Multi-Tenancy Strategy

Database-enforced isolation using PostgreSQL Row Level Security (RLS) ensures complete tenant separation.

RLS Policy Example

-- Enable RLS on workspaces table
ALTER TABLE workspaces ENABLE ROW LEVEL SECURITY;

-- Create policy: Users can only see workspaces in their organization
CREATE POLICY org_isolation ON workspaces 
USING (org_id = current_setting('app.current_org_id')::uuid);

-- Create policy: Workspace members can see workspace assets
CREATE POLICY workspace_member_access ON workspace_assets
USING (
    workspace_id IN (
        SELECT workspace_id 
        FROM workspace_users 
        WHERE user_id = current_setting('app.current_user_id')::uuid
    )
);

Security Features

Authentication

  • JWT token-based auth
  • Role-based access control
  • Multi-factor authentication
  • Session management

Data Protection

  • RLS for tenant isolation
  • Encryption at rest (pgcrypto)
  • TLS for data in transit
  • Audit logging

Infrastructure

  • AWS WAF for DDoS protection
  • VPC isolation
  • Security groups
  • Regular backups

🔥 Critical Security Rules

  • Never bypass RLS in application queries
  • Always set session context (org_id, user_id) before queries
  • No cross-org joins in the application layer
  • Audit all sensitive operations in immutable logs

16. Engineering Guide

First 30 Days Onboarding

  • Understand domain invariants (asset, workspace, transfer)
  • Review database schema and ERD thoroughly
  • Never bypass RLS in queries
  • Prefer database constraints over application logic
  • Preserve history; never overwrite audit trails

Golden Rules

  • No cross-org joins – Always filter by org_id
  • No recursive hierarchy queries – Use ltree operators instead
  • No JSON-based business rules – Use executable pg_cron rules
  • No silent automation – Every action must have an audit log
  • No direct RLS bypass – Always set session context

Development Workflow

  1. Create feature branch from main
  2. Write tests first (TDD approach)
  3. Implement feature with RLS context
  4. Run security audit (automated)
  5. Code review by senior engineer
  6. Merge to main after approval

17. Design Decisions Log

1. Core Platform Scope

Decision: The platform is defined as an Asset Lifecycle Management (ALM) SaaS for IoT-enabled assets across batteries, vehicles, HVAC, and medical equipment. Focus areas include warranty risk, operational efficiency, and lifecycle intelligence rather than generic fleet management.

2. Asset as Source of Truth

Decision: Assets are treated as immutable entities with a unique identity. All telemetry, warranty, claims, transfers, and operational history attach to the asset. Assets can appear in multiple workspaces but are never duplicated.

3. Multi-Tenancy & Data Isolation

Decision: Organizations are strict tenants with isolation enforced at the data layer using RLS. Products are global once approved; assets are scoped to organizations with auditable transfer history.

4. Workspace Model

Decision: Workspaces are observational lenses, not ownership containers. Three types are defined: Actual (physical), Hierarchy (organizational), and Operational (problem-driven). Workspaces define KPI boundaries and collaboration scope.

5. Operational Workspaces

Decision: Operational workspaces are problem-driven and time-bound. Any user with access to at least 10 assets can create one. Assets enter via rules or manual pinning. These workspaces are temporary and can be archived.

6. Product & Global Catalog

Decision: OEMs onboard products and variants locally. ConnectM approves products to a global catalog. Once approved, products become reusable and immutable across all organizations.

7. IoT & Rule Engine

Decision: IoT telemetry is configured at product level for consistency. Rules are categorized into alerts, events, operational KPIs, and reports. Rule evaluation drives asset inclusion, alerts, and warranty logic. Rules are executable (pg_cron), not stored as JSON.

8. Dashboards Strategy

Decision: Three dashboard types: Asset Dashboard (single asset truth), Workspace Dashboard (aggregated KPIs), User KPI Dashboard (accountability). Landing page is always the workspace boundary dashboard.

9. Claims & Warranty Model

Decision: Claims are raised against assets and can be initiated from workspaces. Approval, rejection, and reasoning are fully auditable. Workspace-level claim correlation is a first-class view. Warranty terms are digitized and automatically enforced.

10. Pricing Philosophy

Decision: Pricing is aligned to warranty savings rather than users or devices. Model includes base platform fee, per-asset intelligence fee, and success-based savings share.

11. Collaboration & External Users

Decision: Workspace owners can invite internal and external users (OEM, Financier, 3PL). External users receive scoped access limited to workspace assets. Access is time-bound and revocable.

12. Documentation & Execution

Decision: Conversation history is preserved as a design intent archive. Execution artifacts include Jira-ready user stories, glossary, UI baseline, and decision logs.

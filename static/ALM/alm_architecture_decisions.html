Asset Lifecycle Management (ALM) – Architecture Decision Records

ADR-001: Platform Scope as Asset Lifecycle Management

Status: Accepted | Date: 2026-01

Context

ALM must support IoT-enabled assets across batteries, vehicles, HVAC, and medical equipment while addressing warranty risk and operational efficiency.

Decision

Position the platform as Asset Lifecycle Management, not fleet management, with asset-level truth and lifecycle intelligence.

Consequences

Enables multi-industry scale; avoids fleet-SaaS limitations; increases architectural complexity.

ADR-002: Asset as Immutable Source of Truth

Status: Accepted | Date: 2026-01

Context

Multiple stakeholders interact with the same physical asset over time.

Decision

Treat assets as immutable entities with complete lifecycle history; allow appearance in multiple workspaces.

Consequences

Strong auditability; prevents data duplication; requires careful access control.

ADR-003: Multi-Tenancy with Data-Layer Isolation

Status: Accepted | Date: 2026-01

Context

OEMs, dealers, and financiers require strict data isolation.

Decision

Enforce multi-tenancy at the data layer (e.g., RLS) with organization-scoped assets and auditable transfers.

Consequences

Enterprise-grade security; increased schema and query complexity.

ADR-004: Workspace as Observational Lens

Status: Accepted | Date: 2026-01

Context

Operations teams need problem-centric views rather than static groups.

Decision

Define workspaces as KPI-bound observational lenses, not ownership containers.

Consequences

Supports flexible analysis; avoids hierarchy explosion; requires user education.

ADR-005: Operational Workspaces (Problem-Driven)

Status: Accepted | Date: 2026-01

Context

Temporary investigations are common in warranty and ops workflows.

Decision

Allow any user with access to ≥10 assets to create time-bound operational workspaces with rule-based and manual asset inclusion.

Consequences

Improves collaboration; prevents misuse via eligibility rules; adds lifecycle management overhead.

ADR-006: Product Global Catalog with Approval Gate

Status: Accepted | Date: 2026-01

Context

Products must be reusable across organizations without fragmentation.

Decision

OEMs create products locally; ConnectM approves to global catalog; approved products are immutable.

Consequences

Ensures consistency; introduces governance step.

ADR-007: Rule Engine at Product Level

Status: Accepted | Date: 2026-01

Context

Alerts, events, and KPIs must be consistent across assets of the same product.

Decision

Configure IoT parameters and rules at product level; apply uniformly to all assets.

Consequences

Reduces configuration drift; limits per-asset customization.

ADR-008: Three-Tier Dashboard Strategy

Status: Accepted | Date: 2026-01

Context

Different roles require different views of the same data.

Decision

Define Asset Dashboard (truth), Workspace Dashboard (KPIs), and User KPI Dashboard (accountability).

Consequences

Clear separation of concerns; avoids KPI duplication.

ADR-009: Claims Tied to Assets and Workspaces

Status: Accepted | Date: 2026-01

Context

Warranty claims require evidence and audit trails.

Decision

Bind claims to assets; allow initiation and correlation from workspaces.

Consequences

Improves traceability; adds relational complexity.

ADR-010: Warranty-Savings-Aligned Pricing

Status: Accepted | Date: 2026-01

Context

Traditional per-user or per-device pricing misaligns incentives.

Decision

Adopt pricing based on base platform fee, per-asset intelligence fee, and success-based warranty savings share.

Consequences

Strong value alignment; requires robust savings measurement.

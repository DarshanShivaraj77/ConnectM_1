1. Global Product Catalog

products

  • id (PK, UUID)
  • name (unique, VARCHAR)
  • sku_prefix (VARCHAR)
  • category (VARCHAR)
  • manufacturer (VARCHAR)
  • active (BOOLEAN)
  • created_at (TIMESTAMP)
  • updated_at (TIMESTAMP)

OEM-agnostic master catalog shared across all organizations. Approved products are immutable.

product_variants

  • id (PK, UUID)
  • product_id (FK → products)
  • variant_code (VARCHAR, unique within product)
  • specs (JSONB) — technical specifications
  • pricing (JSONB) — pricing tiers and rules
  • warranty_terms (JSONB) — default warranty configuration
  • created_at (TIMESTAMP)

Extends products with variant-specific attributes like battery capacity, motor power, etc.

2. Tenant & Identity Layer

organizations

  • id (PK, UUID)
  • name (VARCHAR)
  • business_type (ENUM: OEM, Fleet, Dealer, Financier, Service)
  • registration_number (VARCHAR)
  • metadata (JSONB) — branding, config, contact info
  • active (BOOLEAN)
  • created_at (TIMESTAMP)

Root tenant entity. All data is isolated by organization using Row-Level Security (RLS).

users

  • id (PK, UUID)
  • email (VARCHAR, unique)
  • org_id (FK → organizations)
  • role (ENUM: admin, manager, technician, viewer)
  • password_hash (VARCHAR)
  • active (BOOLEAN)
  • last_login (TIMESTAMP)
  • created_at (TIMESTAMP)

Users are scoped to organizations. Cross-org collaboration requires explicit asset transfers.

3. Core Asset Model

assets

  • id (PK, UUID) — immutable, follows asset across transfers
  • product_id (FK → products)
  • variant_id (FK → product_variants)
  • current_org_id (FK → organizations)
  • sku (VARCHAR, unique per org) — organization-specific identifier
  • serial_number (VARCHAR, unique globally)
  • imei_id / sim_id (VARCHAR, encrypted)
  • manufacturing_date (DATE)
  • warranty_expiry (DATE)
  • metadata (JSONB) — custom fields, purchase info, etc.
  • created_at (TIMESTAMP)
  • updated_at (TIMESTAMP)

Central asset registry. Assets are immutable identities that can transfer between organizations.

asset_status

  • id (PK, UUID)
  • asset_id (FK → assets)
  • status (ENUM: operational, maintenance, breakdown, in-transit, decommissioned)
  • location (VARCHAR)
  • assigned_to_workspace (FK → workspaces)
  • assigned_at (TIMESTAMP)
  • notes (TEXT)

Historical record of asset status changes. Current status is latest record by assigned_at.

4. Workspace Hierarchy (6-Level Constraint)

workspaces

  • id (PK, UUID)
  • org_id (FK → organizations)
  • parent_id (FK → workspaces, self-referencing)
  • level (INT, CHECK: 0–6) — enforces max depth
  • type (ENUM: region, hub, depot, fleet, asset)
  • name (VARCHAR)
  • path (LTREE) — hierarchical path for efficient queries
  • metadata (JSONB) — location, config, KPIs
  • created_at (TIMESTAMP)

Flexible hierarchy for organizing assets. Uses PostgreSQL LTREE for efficient tree operations.

workspace_assets

  • id (PK, UUID)
  • workspace_id (FK → workspaces)
  • asset_id (FK → assets)
  • assigned_at (TIMESTAMP)
  • assigned_by (FK → users)

Many-to-many relationship. Assets exist only at lowest hierarchy level (leaf nodes).

workspace_users

  • id (PK, UUID)
  • workspace_id (FK → workspaces)
  • user_id (FK → users)
  • role (ENUM: admin, editor, viewer)
  • assigned_at (TIMESTAMP)

Workspace-level permissions. Workspace cannot exist without at least one admin user.

5. Asset Transfer & Audit

asset_transfers

  • id (PK, UUID)
  • asset_id (FK → assets)
  • from_org_id (FK → organizations)
  • to_org_id (FK → organizations)
  • from_workspace (FK → workspaces)
  • to_workspace (FK → workspaces)
  • status (ENUM: pending, approved, completed, rejected)
  • transfer_reason (TEXT)
  • initiated_by (FK → users)
  • approved_by (FK → users)
  • created_at (TIMESTAMP)
  • completed_at (TIMESTAMP)

Immutable audit trail of asset ownership changes. Warranty and telemetry history follow the asset.

6. Warranty Management

asset_warranty

  • id (PK, UUID)
  • asset_id (FK → assets, unique)
  • start_date (DATE)
  • end_date (DATE)
  • coverage_type (VARCHAR)
  • terms (JSONB) — coverage rules, exclusions
  • created_at (TIMESTAMP)

One-to-one relationship with assets. Warranty follows asset across transfers.

warranty_claims

  • id (PK, UUID)
  • asset_id (FK → assets)
  • claim_number (VARCHAR, unique)
  • issue_type (VARCHAR)
  • description (TEXT)
  • status (ENUM: open, under_investigation, approved, rejected, resolved)
  • evidence (JSONB) — IoT data snapshots, photos, reports
  • submitted_by (FK → users)
  • reviewed_by (FK → users)
  • resolution (TEXT)
  • created_at (TIMESTAMP)
  • resolved_at (TIMESTAMP)

Claims are tied to assets and can be initiated from workspace dashboards.

7. Rule Engine & Analytics (Extensible)

executable_rules

  • id (PK, UUID)
  • org_id (FK → organizations)
  • product_id (FK → products) — apply to all assets of this product
  • asset_id (FK → assets) — optional, for asset-specific rules
  • rule_type (VARCHAR: SOH, speed, geofence, warranty, maintenance)
  • condition (JSONB) — threshold + operator (e.g., {"soc": {"<": 20}})
  • action (ENUM: alert, ticket, webhook, email)
  • action_config (JSONB) — webhook URL, email recipients, etc.
  • active (BOOLEAN)
  • created_at (TIMESTAMP)

Product-level rules apply to all assets. Can be overridden at asset level.

rule_execution_log

  • id (PK, UUID)
  • rule_id (FK → executable_rules)
  • asset_id (FK → assets)
  • status (ENUM: success, failure)
  • trigger_data (JSONB) — snapshot of telemetry that triggered rule
  • action_result (JSONB) — response from webhook, alert ID, etc.
  • executed_at (TIMESTAMP)

Audit log for rule executions. Enables debugging and analytics on rule effectiveness.

telemetry (Time-Series)

  • asset_id (FK → assets)
  • timestamp (TIMESTAMPTZ)
  • params (JSONB) — 70-250 parameters from CAN/MQTT
  • location (POINT) — GPS coordinates

Optimized for time-series queries. Consider TimescaleDB or partitioning for scale.

9. Key Architectural Guarantees

  • Asset Immutability: Asset ID never changes; follows asset across transfers
  • Single Source of Truth: Asset exists at lowest hierarchy level only (leaf nodes)
  • Workspace Constraint: Cannot exist without at least one admin user
  • User Protection: User deletion blocked if they are the last admin of any workspace
  • Multi-Tenancy: Row-Level Security (RLS) enforced via current_org_id
  • Transfer Preservation: Warranty and telemetry history follow the asset
  • Hierarchy Depth: Maximum 6 levels enforced via CHECK constraint
  • Product Consistency: Rules configured at product level apply uniformly to all assets
  • Audit Trail: All transfers, status changes, and rule executions are logged
  • Scalability: Telemetry optimized for millions of rows/day using time-series partitioning

10. Design Principles Summary

  • Asset-Centric: Assets are the immutable source of truth, not organizations or workspaces
  • Observable Workspaces: Workspaces are KPI-bound lenses, not ownership containers
  • Global Catalog: Products approved once, reused across all organizations
  • Rule Engine: Executable, not passive — triggers actions based on real-time telemetry
  • Warranty Intelligence: Claims tied to assets with IoT evidence correlation
  • Transfer Transparency: Full audit trail with stakeholder approval workflow

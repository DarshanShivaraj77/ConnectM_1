Overview

GOCC is a centralized, cloud-connected command platform for EV bus fleet & depot operations, integrating telematics, alerts, analytics, warranty and maintenance lifecycle, plus role-based dashboards across stakeholders.

Multi-tenant
Org isolation via RLS + workspace scoping
Real-time
Kafka/Flink alerts + live fleet monitoring
Lifecycle
Warranty + scheduled maintenance + ticketing

Workspace-centric navigation

Internal users get full modules; Retail users see a restricted set (vehicle list, warranty, maintenance, tickets, reports).

Workspaces model

Workspaces are hierarchical groupings (max 6 levels) that organize assets (buses, batteries, chargers) and permissions (workspace-users), enabling scalable multi-depot and multi-operator management.

Typical hierarchy

  • Region → Zone → Hub → Depot/Dealer → Fleet → Vehicle
  • Same model supports GCC depots and Retail operators

Why it matters

  • Fast filtering for dashboards + reports
  • Delegated access control without custom rules per depot
Workspace Hierarchy Details

The workspace model supports deep hierarchies with:

  • Up to 6 levels of nesting
  • Flexible naming conventions per organization
  • Automatic permission inheritance
  • Cross-workspace reporting capabilities
Permission Management

Role-based access control (RBAC) enables:

  • Granular permissions at workspace level
  • Inherited access from parent workspaces
  • Custom role definitions per organization
  • Audit logging for all permission changes

Platform architecture (high level)

  • Frontend: React/Angular web dashboards + optional mobile apps

Data Storage Strategy

Multi-tier storage approach:

  • PostgreSQL: Master data, transactions, user management
  • TimescaleDB: Time-series telemetry data with automatic compression
  • Redis: Session caching, real-time data buffering
  • S3/MinIO: Object storage for videos, documents, images
  • Elasticsearch: Full-text search and log aggregation

Data model (summary)

Core entities: Organisation, User/Role, Workspace, Asset (Vehicle/Battery/Charger), Telematics, Trips, Alerts, Warranty, Service Jobs, Tickets.

Telematics table pattern (TimescaleDB)

CREATE TABLE vehicle_telemetry (
  ts_utc timestamptz NOT NULL,
  vehicle_id bigint NOT NULL,
  device_id bigint NOT NULL,
  lat numeric(10,7), 
  lon numeric(10,7),
  soc_pct numeric(5,2),
  odo_km numeric(12,3),
  speed_kmph numeric(6,2),
  batt_temp_c numeric(5,2),
  raw_extra jsonb,
  PRIMARY KEY (vehicle_id, ts_utc)
);

-- Create hypertable for time-series optimization
SELECT create_hypertable('vehicle_telemetry', 'ts_utc');

-- Add compression policy (7 days)

  timescaledb.compress,
  timescaledb.compress_segmentby = 'vehicle_id'
);

Core Tables Overview

Primary database tables:

  • organisations - Top-level tenant entities
  • workspaces - Hierarchical groupings
  • users - User accounts and profiles
  • roles - Permission definitions
  • workspace_users - User-workspace assignments
  • vehicles - Vehicle master data
  • batteries - Battery pack information
  • chargers - Charging infrastructure
Operational Tables

Transaction and event tables:

  • trips - Journey records with start/end times
  • alerts - System-generated notifications
  • tickets - Support and maintenance tickets
  • service_jobs - Scheduled and completed services
  • warranty_claims - Warranty submissions and approvals

API specification (OpenAPI)

RESTful API with JWT authentication and role-based access control.

Key endpoints

  • POST /auth/login - User authentication
  • GET /workspaces - List accessible workspaces
  • GET /vehicles - Retrieve vehicle list
  • GET /vehicles/{id}/telemetry - Real-time vehicle data
  • POST /alerts - Create alert
  • GET /reports/fleet-summary - Fleet analytics
Authentication Flow

JWT-based authentication with refresh tokens:

  1. Client sends credentials to /auth/login
  2. Server validates and returns access + refresh tokens
  3. Access token (short-lived, 15 min) used for API calls
  4. Refresh token (long-lived, 7 days) used to obtain new access tokens
  5. Tokens include user roles and workspace permissions
Rate Limiting & Throttling

API rate limits by endpoint type:

  • Authentication: 10 requests/minute
  • Read operations: 100 requests/minute
  • Write operations: 50 requests/minute
  • Bulk operations: 10 requests/minute
  • Real-time telemetry: WebSocket connection (no polling)

Implementation roadmap (deck-ready)

Use the dependency map to sequence delivery. Suggested 16-week V1 with MVP in ~8 weeks.

  • Weeks 1–2: Foundations (IAM, RLS, Workspaces, org setup)
  • Weeks 3–6: Telematics ingest + live fleet + alert engine
  • Weeks 7–8: Depot Ops dashboards + reporting + ticketing basics
  • Weeks 9–12: Warranty + Maintenance + Retail portal
  • Weeks 13–16: Analytics hardening + optional video telematics

Phase 1: Foundation (Weeks 1-2)

Deliverables:

  • Multi-tenant database schema with RLS policies
  • Authentication service with JWT implementation
  • Workspace hierarchy and permission models
  • Organization onboarding workflow
  • Basic admin dashboard

Success criteria: Admin can create orgs, workspaces, and assign users with proper isolation.

Phase 2: Telematics & Monitoring (Weeks 3-6)

Deliverables:

  • Kafka ingestion pipeline for vehicle telemetry
  • TimescaleDB storage with compression policies
  • Real-time fleet map view
  • Alert rule engine (Flink/Kafka Streams)
  • WebSocket API for live updates
  • Basic notification system (email/SMS)

Success criteria: System processes 1000+ vehicles with <1s latency for critical alerts.

Phase 3: Operations (Weeks 7-8)

Deliverables:

  • Depot manager dashboard
  • Fleet performance reports (utilization, efficiency)
  • Ticketing system for maintenance requests
  • Trip history and analytics
  • Driver assignment module

Success criteria: Depot managers can monitor fleet, create tickets, and generate reports.

Phase 4: Lifecycle Management (Weeks 9-12)

Deliverables:

  • Warranty management system
  • Scheduled maintenance tracker
  • Service history logging
  • Retail operator portal (restricted views)
  • Spare parts inventory integration

Success criteria: Complete warranty and maintenance lifecycle tracking operational.

Phase 5: Advanced Features (Weeks 13-16)

Deliverables:

  • Advanced analytics dashboard (ML-based insights)
  • Predictive maintenance algorithms
  • Video telematics integration (optional)
  • Mobile app for drivers
  • API documentation and developer portal
  • Performance optimization and load testing

Success criteria: System handles 5000+ vehicles with predictive insights and video storage.
